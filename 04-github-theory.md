# GitHub_Theory

## 1. Issue（イシュー）の基本

タスクはすべてIssueから始まります。「GTD（タスク管理）」の感覚でどんどん作りましょう。

- **種類:** 「不具合（Bug）」か「機能追加（Feature）」の2つ

- **管理:**
  - **Open:** 未完了
  - **Closed:** 完了
  - **TBD:** 未定という意味（To Be Determined）ここに内容を書く

- **運用:**
  - **Assignees:** 担当者を設定する
  - **#（シャープ）+ 番号:** issueにはuniqueに番号がつく、「#6の件ですが…」のように会話で使う
  - **Insights:** 誰がどれだけ動いているか可視化される機能

---

## 2. ブランチとPush/Pull

コードを書くときは、必ず「自分の枝（ブランチ）」で作業します。

- **命名規則:** `issue番号_タイトル略称`
  - 例：`003_fix_branch_name`

- **基本の動き:**
  - **Push:** 各ブランチで行う
  - **Pull:** mainブランチで行う（最新をローカルに取り込む）

- **同期のコマンド:**
リモートリポジトリが進化して自分のリポジトリと内容が変わっていくので
たまに最新をpullしてくる
  - `git fetch`: リモートの最新状態を確認
  - `git pull origin main`: リモートのmainを自分のローカルに反映

---

## 3. Pull Request（プルリク / PR）

コードをmainに合流させるための「依頼」です。

- **構成:**
  - PRにもuniqueに番号がつく
  - 説明欄に「関連するIssue番号」を貼る
  - **Reviewer（レビュアー）:** 評価する人、レビューする人
  - **Reviewee（レビューイー）:** 評価される人、レビューを受ける人

- **AI活用:** CodeRabbit（コードラビット）が自動で添削・コメントをくれる

---

## 4. レビューのやり方

効率的なレビューのために「Start a review」を活用しましょう。

> インラインコメントの手順

### レビューのステータス

|**ステータス**|**意味**|
|---|---|
|**Comment**|質問や称賛「承認」も「却下」もしない|
|**Approve**|アプルーブ 承認。「マージしてOK！」|
|**Request changes**|リクエストチェンジ 変更要求「直さないとマージ不可」という強い意思表示|

---

## 5. 後片付けと管理

不要になったファイルやIssueは、後で見ても理由がわかるようにラベルを貼ります

- **duplicate:** ディプリケイト重複（同じ内容がすでにある）
- **wontfix:** 修正しない（もう復活させない、必要なくなった）

---

## 6. 作業の最後

レビュアーにApprove（承認）をもらったら

1. **Merge Request**（マージ）を実行
2. ターミナル（Ghostty等）で `git pull origin main` を実行し、手元のPCも最新状態にする
